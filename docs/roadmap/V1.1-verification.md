# v1.1.0 — SPI Foundations for Cloud / Multi-Tenant / Code-Agent — Release Verification

**State:** Released (4 of 4 P0 items complete)
**Scope frozen at:** 2026-04-25
**Released at:** 2026-04-25
**Canonical scope driver:** `.plans/V1.1-SPI-FOUNDATIONS.md`
**SOT row:** `.plans/VERSION-STATUS-SOT.md` § Canonical Status Table — `v1.1.0`

## Goal

Close the v1.1 release. v1.0 GA delivered the public framework surface; v1.1 builds the
four cross-cutting SPI seams that AgentCode (line B) and any future cloud / multi-tenant
deployment need before they can integrate without forking Kairo:

1. **F1 ExecutionSandbox SPI** — pluggable shell-execution backends (the v1.0 `BashTool`
   hard-coded `ProcessBuilder` is now a default implementation, not the contract).
2. **F2 Workspace SPI** — explicit working-directory context replacing the implicit
   `System.getProperty("user.dir")` ambient assumption baked into every file tool.
3. **F3 TenantContext** — passive multi-tenant identity propagation across the runtime
   (audit / OTel / events / tools / bridge frames inherit the binding without any
   method-signature change).
4. **F4 Bridge Protocol** — minimal bidirectional RPC for companion clients (TUI / Web /
   IDE plugin) to drive the agent runtime over the existing WS transport.

All four SPIs are pure additions on top of v1.0 GA's frozen surface. No existing public
method signatures change; no v1.0 contract is broken. japicmp confirms.

## Scope & Evidence (four P0 items + F5 wrap-up)

### F1 — ExecutionSandbox SPI — **DONE**

**SPI package:** `io.kairo.api.sandbox.*` (in `kairo-api`)
**Default implementation:** `LocalProcessSandbox` (in `kairo-tools`)

**Files:**
- `kairo-api/src/main/java/io/kairo/api/sandbox/ExecutionSandbox.java` — single-method SPI: `SandboxHandle start(SandboxRequest)`
- `kairo-api/src/main/java/io/kairo/api/sandbox/SandboxRequest.java` — record + Builder, defaults: `timeout = 30s`, `maxOutputBytes = 1 MiB`, `readOnly = false`
- `kairo-api/src/main/java/io/kairo/api/sandbox/SandboxHandle.java` — AutoCloseable; hot `Flux<SandboxOutputChunk>`; cached `Mono<SandboxExit>`; idempotent `cancel()` / `close()`
- `kairo-api/src/main/java/io/kairo/api/sandbox/SandboxOutputChunk.java` — sealed: `Stdout(byte[])` / `Stderr(byte[])`
- `kairo-api/src/main/java/io/kairo/api/sandbox/SandboxExit.java` — record: `exitCode` / `signal` / `timedOut` / `truncated`
- `kairo-api/src/main/java/io/kairo/api/sandbox/package-info.java` — package contract
- `kairo-tools/src/main/java/io/kairo/tools/sandbox/LocalProcessSandbox.java` — default impl wrapping `ProcessBuilder`
- `kairo-tools/src/main/java/io/kairo/tools/sandbox/tck/ExecutionSandboxTCK.java` — abstract JUnit 5 contract kit

**Behavior contract (TCK-enforced, 8 scenarios):**

| # | Scenario | Pass condition |
|---|---|---|
| 1 | Successful exit | exit code 0, full output captured |
| 2 | Non-zero exit | exit code surfaced verbatim |
| 3 | Timeout fires | process killed; `exit().timedOut() == true`; `exit().exitCode() == -1` |
| 4 | Output exceeds `maxOutputBytes` | output truncated; `exit().truncated() == true` |
| 5 | `cancel()` is idempotent | second invocation is a no-op; backend signal surfaced |
| 6 | `readOnly == true` | rejects writes to `workspaceRoot` |
| 7 | Concurrent `start()` calls | isolated; no shared state leak |
| 8 | `close()` releases resources | idempotent |

**`BashTool` integration:** Resolves the active sandbox from `ToolContext`; falls back
to `LocalProcessSandbox` when none is bound. Public method signatures unchanged. Existing
single-process call sites observe zero behavior change.

**ADR:** `docs/adr/ADR-025-execution-sandbox-spi.md` — Status: Accepted.

### F2 — Workspace SPI — **DONE**

**SPI package:** `io.kairo.api.workspace.*` (in `kairo-api`)
**Default implementation:** `LocalDirectoryWorkspaceProvider` (in `kairo-core`)

**Files:**
- `kairo-api/src/main/java/io/kairo/api/workspace/Workspace.java` — `id()` / `root()` / `kind()` / `metadata()` + `cwd()` no-dependency static factory
- `kairo-api/src/main/java/io/kairo/api/workspace/WorkspaceProvider.java` — `acquire(WorkspaceRequest)` / `release(String workspaceId)`
- `kairo-api/src/main/java/io/kairo/api/workspace/WorkspaceKind.java` — enum: `LOCAL` (ships) / `REMOTE_GIT` / `EPHEMERAL` (reserved for v1.3)
- `kairo-api/src/main/java/io/kairo/api/workspace/WorkspaceRequest.java` — record: `hint` / `tenant` / `writable`
- `kairo-api/src/main/java/io/kairo/api/workspace/CwdWorkspace.java` — package-private impl for `Workspace.cwd()`
- `kairo-api/src/main/java/io/kairo/api/workspace/package-info.java` — package contract

**Relative-path resolution rule (uniform across 5 file tools — `Read` / `Write` / `Edit`
/ `Glob` / `Grep`):**

1. `Path.isAbsolute(arg)` → use verbatim.
2. Else → `currentWorkspace().root().resolve(arg).normalize()`.
3. Resolved path MUST `startsWith(root)` after normalisation; otherwise rejected with
   `IllegalArgumentException` (defence-in-depth path-traversal guard).

**Backward compatibility:** When `ToolContext` does not bind a workspace,
`currentWorkspace()` returns `Workspace.cwd()` — a `LOCAL` workspace rooted at
`Path.of("").toAbsolutePath()`. This is exactly what file tools used in v1.0. Existing
single-workspace deployments observe **zero** behavior change.

**`BashTool` integration:** Constructs `SandboxRequest.workspaceRoot()` from
`currentWorkspace().root()` (binds F1 ↔ F2).

**ADR:** `docs/adr/ADR-026-workspace-spi.md` — Status: Accepted.

### F3 — TenantContext — **DONE**

**SPI package:** `io.kairo.api.tenant.*` (in `kairo-api`)
**Default implementation:** `ThreadLocalTenantContextHolder` + `ReactorTenantContextPropagator` (in `kairo-core`)

**Files:**
- `kairo-api/src/main/java/io/kairo/api/tenant/TenantContext.java` — record with `SINGLE` sentinel + `ATTR_TENANT_ID` / `ATTR_PRINCIPAL_ID` constants
- `kairo-api/src/main/java/io/kairo/api/tenant/TenantContextHolder.java` — `current()` + `bind(ctx) → Scope` propagation seam + `NOOP` opt-out sentinel
- `kairo-api/src/main/java/io/kairo/api/tenant/package-info.java` — package contract

**Propagation rules:**

1. **Boundary code binds; downstream reads.** HTTP filters, message consumers, and
   scheduled-job entry points are the only callers that should invoke
   `holder.bind(ctx)`. Internal code reads via `holder.current()` and never re-binds.
2. **Try-with-resources is mandatory for `bind()`.** A leaked scope corrupts the
   thread-local stack for subsequent requests. `Scope.close()` is safe to call multiple
   times.
3. **Reactor pipelines inherit at subscribe time.** The propagator captures the binding
   when the pipeline is subscribed and re-binds it on each downstream signal.
4. **`SINGLE` is the implicit default.** Read sites that observe no binding see
   `SINGLE = ("default", "anonymous", {})`. Existing single-tenant deployments observe
   zero behavior change.

**Five passive consumption sites (read-only — no method signatures change):**

| Sink | Projection |
|---|---|
| `KairoEventBus.publish(...)` | Attaches `tenant.id` + `tenant.principal` to event attributes if not already present |
| `KairoEventOTelExporter` | Emits `kairo.tenant.id` + `kairo.tenant.principal` on every span / log record |
| `SecurityEventSink.record(...)` | Adds `tenant.id` to the audit event attribute map |
| `ToolContext.tenant()` | Accessor exposed for tenant-aware tools (e.g., tenant-scoped DB tools) |
| `BridgeMeta.tenant` (F4) | Bridge frames inherit the propagated tenant on inbound dispatch |

**Explicit non-goals (deferred to v1.2 per `.plans/V1.2-DISTRIBUTED.md` D1):**
- Quota enforcement (per-tenant token / cost / request-rate limits).
- Physical isolation (per-tenant DB schemas, per-tenant Kafka topic prefixes).
- Cross-tenant routing (`Channel` adapters fanning a single inbound message to multiple
  tenant-scoped agents — v1.3).
- Tenant-aware `Resilience4j` integrations (alongside v1.2 quota work).
- Authentication / authorization (Spring Security upstream owns this).

**ADR:** `docs/adr/ADR-027-tenant-context.md` — Status: Accepted.

### F4 — Bridge Protocol — **DONE**

**SPI package:** `io.kairo.api.bridge.*` (in `kairo-api`)
**Default transport:** `WebSocketBridgeServer` + `KairoBridgeWebSocketHandler` (in `kairo-event-stream-ws`)

**Files:**
- `kairo-api/src/main/java/io/kairo/api/bridge/BridgeRequest.java` — record: `op` / `payload (Map<String,Object>)` / `meta (BridgeMeta)`
- `kairo-api/src/main/java/io/kairo/api/bridge/BridgeResponse.java` — record: `status (int)` / `payload (Map<String,Object>)` + factory helpers (`ok` / `notFound` / `error`)
- `kairo-api/src/main/java/io/kairo/api/bridge/BridgeMeta.java` — record: `requestId` / `tenant` / `timestamp` / `attributes`
- `kairo-api/src/main/java/io/kairo/api/bridge/BridgeRequestHandler.java` — `Mono<BridgeResponse> handle(BridgeRequest)` dispatcher SPI
- `kairo-api/src/main/java/io/kairo/api/bridge/BridgeServer.java` — `start()` / `stop()` / `endpoint()` lifecycle SPI
- `kairo-api/src/main/java/io/kairo/api/bridge/package-info.java` — package contract with op catalog table
- `kairo-event-stream-ws/src/main/java/io/kairo/eventstream/ws/WebSocketBridgeServer.java` — default `BridgeServer` impl
- `kairo-event-stream-ws/src/main/java/io/kairo/eventstream/ws/KairoBridgeWebSocketHandler.java` — Spring WebFlux handler
- `kairo-event-stream-ws/src/test/java/io/kairo/eventstream/ws/KairoBridgeWebSocketHandlerTest.java` — 8 integration tests

**Frozen 5-op catalog (v1.1):**

| op | Inbound payload | Outbound payload | Purpose |
|---|---|---|---|
| `agent.run` | `{ agent, input, ... }` | `{ runId, status }` | Kick off an agent invocation |
| `agent.cancel` | `{ runId }` | `{ cancelled }` | Terminate the active invocation |
| `agent.status` | `{ runId }` | `{ runId, phase }` | Query session state |
| `tool.approve` | `{ toolCallId, decision, reason }` | `{ acked }` | Accept / reject a risky tool execution (feeds `UserApprovalHandler`) |
| `workspace.list` | `{}` | `{ workspaces: [{id, kind, ...}, ...] }` | Enumerate workspaces visible to the principal |

**Wire format:** UTF-8 JSON envelopes mounted on `/ws/bridge` (separate from event-stream
WS). Inbound `{requestId, op, payload, meta}` → outbound `{requestId, status, payload}`.
Missing `requestId` is synthesised by the server and reflected on the response so the
client can still correlate.

**Status code mapping:**

| Status | Cause | Session impact |
|---|---|---|
| 200 | Handler returned `BridgeResponse.ok(...)` | Session stays open |
| 400 | Malformed JSON inbound frame | Session stays open |
| 404 | Unknown op (handler returned `BridgeResponse.notFound(op)`) | Session stays open |
| 500 | Handler threw / returned null | Session stays open |
| WS close `4503 "bridge-stopped"` | `BridgeServer.stop()` flipped the lifecycle gate | New sessions rejected; in-flight sessions complete naturally |

**Application-level errors NEVER close the session.** Only transport-level failures and
explicit shutdown drop the connection. A misbehaving client can spam malformed frames
and well-formed ones still get served.

**Plan deviation captured in ADR-028:** Payload typed as `Map<String, Object>` instead
of `JsonNode` (per F4 spec sketch in `.plans/V1.1-SPI-FOUNDATIONS.md`) to keep
`kairo-api` Jackson-free, consistent with the `KairoEvent.attributes` precedent and
ADR-023's dependency-hygiene rule for stable types.

**Tests:** 8 integration tests in `KairoBridgeWebSocketHandlerTest`:

| # | Test | Asserts |
|---|---|---|
| 1 | `agentRun_dispatchesPayload_andCorrelatesResponse` | op 1 happy path + requestId correlation |
| 2 | `agentCancel_returnsOk_evenWhenAlreadyTerminated` | op 2 happy path |
| 3 | `agentStatus_synthesizesRequestId_whenClientOmits` | op 3 + server-synthesized requestId |
| 4 | `toolApprove_propagatesDecisionToHandler` | op 4 + decision propagation |
| 5 | `workspaceList_returnsArrayPayload` | op 5 + array payload shape |
| 6 | `unknownOp_resolvesToFourOhFour` | 404 error path |
| 7 | `malformedFrame_resolvesToFourHundred` | 400 error path |
| 8 | `handlerThrows_resolvesToFiveHundred` | 500 error path |

**ADR:** `docs/adr/ADR-028-bridge-protocol.md` — Status: Accepted.

## Anti-goals (reaffirmed — explicitly NOT shipped in v1.1)

Locked by `.plans/V1.1-SPI-FOUNDATIONS.md` § "不在范围内（明确拒绝的诱惑）":

- ❌ `DockerSandbox` / `FirecrackerSandbox` (container / micro-VM backends) — **v1.2**.
- ❌ `RemoteHttpSandbox` (gRPC / HTTP execution backend) — **v1.2**.
- ❌ Tenant quota enforcement (per-tenant token / cost / rate limits) — **v1.2**
  (`.plans/V1.2-DISTRIBUTED.md` D1).
- ❌ Tenant physical isolation (per-tenant DB schemas / Kafka topic prefixes) —
  **v1.2**.
- ❌ `RemoteGitWorkspaceProvider` (JGit / GitHub-API-backed checkout) — **v1.3**.
- ❌ `EphemeralWorkspaceProvider` (sandbox-mounted scratch directories) — **v1.3**.
- ❌ Bridge file-transfer protocol (large payloads stay out of band) — **v1.3**.
- ❌ Bridge protocol-version field (schemaless envelope makes it unnecessary).
- ❌ Bridge transport-level authentication (Spring Security upstream owns it).
- ❌ Server-pushed bridge frames (`KairoEventBus` event-stream covers that).
- ❌ Any UI / TUI tool work — that is line B (AgentCode), not Kairo.
- ❌ Self-Evolution v0.9 advance — gated on v1.2.
- ❌ Realtime / RL — **v2.0**.
