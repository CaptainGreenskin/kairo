# v0.10 Expert Team Kickoff — Orchestration MVP

> **Status**: Frozen scope — landed alongside the v0.10 Core Refactor Waves as a companion sub-milestone (v0.10 Expert Team MVP).
>
> **Canonical ADRs**: `docs/adr/ADR-015-expert-team-orchestration.md` (pipeline), `docs/adr/ADR-016-coordinator-spi.md` (SPI shape).

## Why this sub-milestone

Expert Team was deferred repeatedly (v0.6 → v0.7 → v0.8 → v0.9 "Deferred from this round") because earlier versions needed stable execution primitives first. With v0.8 DurableExecution / ResourceConstraint delivered and v0.10 Core Refactor Waves shipping `KairoEventBus` + capability configs, the runway was finally clear. This kickoff captures the minimal MVP that:

1. Ships a **dedicated `kairo-expert-team` module** — clean boundary separate from `kairo-multi-agent`.
2. Exposes exactly **two SPIs** (`TeamCoordinator`, `EvaluationStrategy`) plus a TCK, honoring *SPI-Earned* while acknowledging the infrastructure-extensibility exception.
3. Delivers a **Spring Boot starter** (`kairo-spring-boot-starter-expert-team`) with opt-in auto-configuration.
4. Proves six end-to-end scenarios under `@Tag("integration")` → Failsafe.

## Inputs / preconditions

- v0.10 Core Refactor Waves — `KairoEventBus` + `KairoEvent` envelope available; team events live on `KairoEvent.DOMAIN_TEAM`.
- v0.8 ResourceConstraint — reused via `TeamResourceConstraint.unbounded()` default; feedback-loop rounds are bounded by `TeamConfig#maxFeedbackRounds`.
- ADR-015 and ADR-016 already merged (see `docs/adr/`).

## Scope (P0)

| Capability | Shape | Home |
|---|---|---|
| `TeamCoordinator` SPI | `Mono<TeamResult> execute(TeamExecutionRequest, Team)` | `kairo-api` |
| `EvaluationStrategy` SPI | `Mono<EvaluationVerdict> evaluate(EvaluationContext)` | `kairo-api` |
| `TeamConfig` / `TeamExecutionRequest` / `TeamResult` / `TeamStatus` / `RiskProfile` / `EvaluatorPreference` / `PlannerFailureMode` | Value types | `kairo-api` |
| `TeamEventType` + team domain events | Emitted on `KairoEvent.DOMAIN_TEAM` | `kairo-api` |
| `ExpertTeamCoordinator` | Default pipeline impl (Planning → Generating → Evaluating → Terminal) | `kairo-expert-team` |
| `SimpleEvaluationStrategy` / `AgentEvaluationStrategy` | Deterministic rubric + agent-backed evaluator with crash → `REVIEW_EXCEEDED` mapping | `kairo-expert-team` |
| `DefaultPlanner` | Role-per-agent plan with FAIL_FAST default | `kairo-expert-team` |
| TCK | `NoopMessageBus`, `RecordingEventBus`, `StubAgent` under `io.kairo.expertteam.tck` | `kairo-expert-team` (test-jar) |
| Spring Boot starter | `ExpertTeamAutoConfiguration` + `ExpertTeamProperties` — opt-in via `kairo.expert-team.enabled=true` | `kairo-spring-boot-starter-expert-team` |

## State machine (MVP)

```
IDLE → PLANNING → GENERATING → EVALUATING → { COMPLETED | FAILED | DEGRADED | TIMEOUT }
                       ↑               ↓
                       └── feedback ───┘   (bounded by TeamConfig#maxFeedbackRounds)
```

Semantics (per ADR-015):
- **Planner failure** → `PlannerFailureMode.FAIL_FAST` default → `TeamStatus.FAILED`.
- **Evaluator crash** (via `AgentEvaluationStrategy`) → `VerdictOutcome.REVIEW_EXCEEDED`:
  - `RiskProfile.LOW` → `TeamStatus.DEGRADED` + warning + last `TEAM_COMPLETED` event.
  - `RiskProfile.MEDIUM|HIGH` → `TeamStatus.FAILED` + last `TEAM_FAILED` event.
- **Team timeout** → `TeamStatus.TIMEOUT`; partial outcomes preserved; `finalOutput()` empty; last event `TEAM_TIMEOUT`.
- **Event domain isolation** → every coordinator-emitted event carries `KairoEvent.DOMAIN_TEAM`; none leak into `DOMAIN_EXECUTION` / `DOMAIN_EVOLUTION` / `DOMAIN_SECURITY`.

## SPI contract guardrails

- `TeamCoordinator` must return a `TeamResult` whose `status()` is one of `{COMPLETED, FAILED, DEGRADED, TIMEOUT}` — never `RUNNING`/null.
- `EvaluationStrategy` errors are **not** mapped by the coordinator; use `AgentEvaluationStrategy` (or equivalent wrapper) to translate exceptions into `VerdictOutcome.REVIEW_EXCEEDED` when crash tolerance is required.
- All team events published during a run share one run identifier (`request.requestId()`).

## Module boundary invariant

`io.kairo.expertteam` source tree may only reference:

```
io.kairo.api.*, io.kairo.expertteam.*, java.*, javax.*, org.slf4j.*, reactor.*
```

Any reference to `io.kairo.multiagent`, `io.kairo.mcp`, `io.kairo.evolution`, `io.kairo.observability`, `io.kairo.tools`, `io.kairo.spring` is a boundary violation. Enforced by an integration test (see verification doc).

## Starter activation

- `@ConditionalOnClass(ExpertTeamCoordinator.class)` — classpath gate.
- `@ConditionalOnProperty(prefix="kairo.expert-team", name="enabled", havingValue="true", matchIfMissing=false)` — **installing the starter is not enough**; users must opt in explicitly, consistent with ADR-015's stance that expert-team orchestration is advanced and policy-sensitive.
- Beans wired when enabled: `simpleEvaluationStrategy`, `agentEvaluationStrategy` (stub if no `AgentInvoker` present), `defaultExpertTeamPlanner`, `expertTeamCoordinator`.
- User-provided `TeamCoordinator` bean wins via `@ConditionalOnMissingBean(TeamCoordinator.class)`.

## Non-goals (deferred)

- Parallel / graph planners — MVP is sequential per ADR-015.
- Tool-augmented evaluators beyond the agent-invoker seam.
- Cross-team handoff / nested teams.
- Advanced reasoning strategies (ToT / MCTS) — remain v1.0 candidates.
- Channel SPI integration (DingTalk / IM surfaces) — v0.9 gap-only stays scoped.
