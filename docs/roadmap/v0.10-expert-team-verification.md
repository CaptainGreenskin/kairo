### Pre-existing test fixes picked up during the release gate

Two unrelated tests were fixed during the release gate so the gate could be fully green:

- `kairo-core/src/test/java/io/kairo/core/integration/AgentPerformanceIT.java::buildAgent()` — missing `.modelName("mock-model")` on the `AgentConfig.builder()` chain caused `ModelConfig.builder().build()` to throw `NPE("model must not be null")` in 3 IT methods; added the setter.
- `kairo-spring-boot-starter-core/src/test/java/io/kairo/spring/NegativeAutoConfigTest.java::contextFailsWithoutApiKey` — assertion expected `IllegalStateException("API key not configured")` but `ModelProviderUtils.validateApiKey` throws `IllegalArgumentException("<provider> apiKey cannot be null or blank")`; updated the expected type and message.

Note on `-Pintegration-tests` + JaCoCo: the profile scopes Surefire to `groups=integration`, so unit tests do not execute in that invocation and the aggregated coverage drops below the `jacoco.line.threshold=0.50` gate. JaCoCo enforcement remains attached to the canonical `mvn clean verify` path; IT runs are for behavioural coverage.

## Scenario coverage (ADR-015 MVP)

`kairo-expert-team/src/test/java/io/kairo/expertteam/integration/ExpertTeamMvpIT.java` exercises six behavioural scenarios plus a module-boundary guard:

| # | Test | What it proves |
|---|---|---|
| 1 | `happyPath_multiAgentPlanReachesCompleted` | Three-agent sequential plan (scribe/critic/editor) reaches `TeamStatus.COMPLETED`. Each agent invoked exactly once; event stream contains three `STEP_ASSIGNED` + three `STEP_COMPLETED` + three `EVALUATION_RESULT`; starts on `TEAM_STARTED` and ends on `TEAM_COMPLETED`. |
| 2 | `feedbackLoop_reviseTwiceThenPassRetainsArtifactAndAttempts` | `EvaluationStrategy` returns REVISE / REVISE / PASS. Step retains attempts=3; agent invoked 3×; three `EVALUATION_STARTED` + three `EVALUATION_RESULT`; terminal `TEAM_COMPLETED` — feedback loop is bounded by `TeamConfig#maxFeedbackRounds`. |
| 3a | `evaluatorCrash_mediumRiskTerminatesFailed` | Evaluator throws inside `AgentEvaluationStrategy` → `VerdictOutcome.REVIEW_EXCEEDED`. Under `RiskProfile.MEDIUM` → `TeamStatus.FAILED`, last event `TEAM_FAILED`. |
| 3b | `evaluatorCrash_lowRiskDegradesWithWarning` | Same crash semantics under `RiskProfile.LOW` → `TeamStatus.DEGRADED` with non-empty warnings; last event `TEAM_COMPLETED`. |
| 4 | `teamTimeout_slowAgentYieldsTimeoutStatusWithPartialOutcomes` | `StubAgent` sleeps 5s against a 200 ms `teamTimeout` → `TeamStatus.TIMEOUT`; warnings populated; `finalOutput()` empty; last event `TEAM_TIMEOUT`. |
| 5 | `eventDomainIsolation_onlyTeamDomainEventsEmitted` | Every event published by the coordinator sits on `KairoEvent.DOMAIN_TEAM`; none leak into `DOMAIN_EXECUTION`, `DOMAIN_EVOLUTION`, or `DOMAIN_SECURITY`. |
| 6 | `moduleBoundary_expertTeamDoesNotReferenceForeignModules` | Walks `src/main/java/io/kairo/expertteam` and asserts no file references `io.kairo.multiagent`, `io.kairo.mcp`, `io.kairo.evolution`, `io.kairo.observability`, `io.kairo.tools`, or `io.kairo.spring`. Allowed deps are `io.kairo.api.*`, `io.kairo.expertteam.*`, `java.*`, `javax.*`, `org.slf4j.*`, `reactor.*`. |

## What Landed (high-signal)

### Module: `kairo-expert-team`

- **SPIs** (in `kairo-api`): `TeamCoordinator`, `EvaluationStrategy` plus supporting value types (`TeamConfig`, `TeamExecutionRequest`, `TeamResult`, `TeamStatus`, `RiskProfile`, `EvaluatorPreference`, `PlannerFailureMode`, `EvaluationVerdict`, `TeamEventType`, `TeamResourceConstraint`).
- **Coordinator**: `ExpertTeamCoordinator` — deterministic Planning → Generating → Evaluating → Terminal pipeline. Emits exclusively on `KairoEvent.DOMAIN_TEAM`.
- **Evaluation**: `SimpleEvaluationStrategy` (deterministic rubric) + `AgentEvaluationStrategy` (agent-invoker seam; maps crashes to `VerdictOutcome.REVIEW_EXCEEDED`).
- **Planner**: `internal/DefaultPlanner` — role-per-agent sequential plan, `PlannerFailureMode.FAIL_FAST` default.
- **TCK** (`io.kairo.expertteam.tck`): `TeamCoordinatorTCK`, `EvaluationStrategyTCK`, plus reusable fixtures `RecordingEventBus`, `NoopMessageBus`, `StubAgent`.

### Module: `kairo-spring-boot-starter-expert-team`

- `ExpertTeamAutoConfiguration` — `@ConditionalOnClass(ExpertTeamCoordinator.class)` + `@ConditionalOnProperty("kairo.expert-team.enabled"=true, matchIfMissing=false)` + `@EnableConfigurationProperties(ExpertTeamProperties.class)`.
- Beans: `simpleEvaluationStrategy`, `agentEvaluationStrategy` (stub via `ObjectProvider<AgentInvoker>`), `defaultExpertTeamPlanner`, `expertTeamCoordinator` (`@ConditionalOnMissingBean(TeamCoordinator.class)`).
- `ExpertTeamProperties` — `enabled`, `defaultRiskProfile`, `defaultMaxFeedbackRounds`, `defaultTeamTimeout`, `defaultEvaluatorPreference`.
- Aggregator `kairo-spring-boot-starter` re-exports the new starter for parity; deprecation messaging unchanged.

### BOM

- `kairo-bom` publishes `kairo-expert-team` and `kairo-spring-boot-starter-expert-team` at `${revision}`, aligned with the rest of the v0.10 train.

## Known Follow-ups (explicit non-completion)

1. **Parallel / graph planners** beyond the sequential default — tracked for a post-v0.10 PR once sequential usage stabilizes.
2. **Channel SPI glue** (IM-driven Expert Team triggers, e.g. DingTalk) — remains v0.9 gap-only + future platform work.
3. **Cross-team handoff / nested teams** — out of MVP scope; revisit after v1.0 stabilization feedback.
4. **Tool-augmented evaluators** beyond the current `AgentEvaluationStrategy.AgentInvoker` seam.
5. **Deprecation follow-through**: once downstream adopters migrate, legacy `io.kairo.api.task.*` / `TeamScheduler` retirement (already marked `@Deprecated(forRemoval=true)` in v0.10 Core Refactor) can be physically deleted.
