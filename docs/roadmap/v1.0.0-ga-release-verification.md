Close the v1.0 GA release. RC1 froze the SPI shape; RC2 closed the docs/examples
gap; this GA wave delivers the enterprise-security capabilities that turn a
"working framework" into something an audited org can adopt: PII redaction,

becomes a stable target for downstream consumers.

## Scope & Evidence (six P0 items)

### B3C.1 — PII redaction policy — **DONE**

**Module:** `kairo-security-pii` (new, reactor module #26)
**Files:**
- `kairo-security-pii/src/main/java/io/kairo/security/pii/PiiRedactionPolicy.java`
- `kairo-security-pii/src/main/java/io/kairo/security/pii/PiiRedactionConfig.java`
- `kairo-security-pii/src/main/java/io/kairo/security/pii/PiiPattern.java`

Implements **the existing `GuardrailPolicy` SPI** (no new abstraction — see
ADR-024 for the deliberate pivot from "new `PiiRedactor` SPI" in plan text to
"stock `GuardrailPolicy`"). Composes with any other guardrail via
`GuardrailChain`.

**Patterns covered (regex-based, opt-in via `PiiRedactionConfig`):** EMAIL,
US_PHONE, CREDIT_CARD, SSN, API_KEY, JWT.

**Behavior:**

| Payload kind | Action | Result |
|---|---|---|
| `GuardrailPayload.ToolOutput` (matches PII)  | `MODIFY` | Rebuilt with redacted text + preserved `toolUseId`/metadata |
| `GuardrailPayload.ModelOutput` (matches PII) | `MODIFY` | `TextContent` + `ThinkingContent` redacted; `ToolUseContent` passes through |
| `ToolInput` / `ModelInput`                   | `ALLOW`  | Out of scope — PII redaction is an output-side concern |
| Phase not in config                          | `ALLOW`  | Skipped (default config: `POST_TOOL` + `POST_MODEL`)  |
| No PII match                                 | `ALLOW`  | Pass-through |

**Tests:** `PiiRedactionPolicyTest` — 15/15 green (per-pattern positive,
multi-pattern composition, phase gating, ToolOutput round-trip with metadata
preservation, ModelOutput Content-walk, factory selection, name+order
exposure, empty/null no-op).

### B3C.2 — Audit log framework — **DONE**

**Files:**
- `kairo-security-pii/src/main/java/io/kairo/security/audit/JdbcAuditEventSink.java`
- `kairo-security-pii/src/main/resources/db/migration/V1__create_kairo_audit_table.sql`

Implements **the existing `SecurityEventSink` SPI** (no new abstraction).
Append-only `kairo_audit` table managed by Flyway.

**Schema:**

```
id              BIGINT AUTO_INCREMENT PRIMARY KEY
event_id        VARCHAR(64)          -- UUID, one per record() call
event_type      VARCHAR(64)          -- SecurityEventType.name()
agent_name      VARCHAR(255)
target_name     VARCHAR(255)
phase           VARCHAR(32)          -- nullable
policy_name     VARCHAR(255)
reason          VARCHAR(1024)        -- truncated at 1024 chars
attributes_json CLOB                 -- String.valueOf(attrs); nullable
created_at      TIMESTAMP NOT NULL

INDEX idx_kairo_audit_created_at  (created_at)
INDEX idx_kairo_audit_agent       (agent_name)
INDEX idx_kairo_audit_event_type  (event_type)
```

**Design choices:**
- Plain `javax.sql.DataSource` (no Spring) → keeps the module lean
- SPI contract obeyed: `SQLException` is logged via slf4j and **swallowed** so
  audit persistence cannot break the request path
- `String.valueOf(attrs)` for attributes (no Jackson dep); hosting apps that
  need queryable JSON should compose a custom sink

**Tests:** `JdbcAuditEventSinkTest` — 5/5 green
(`singleEventIsPersisted`, `manyEventsAreAppended` with 25 inserts,
`nullPhaseAndAttributesAreToleratedAsNullColumns`, `overlongReasonIsTruncated`
2000→1024, `sqlFailureIsSwallowed` with bad DataSource).

### B3C.3 — Per-run compliance report — **DONE**

**Files:**
- `kairo-security-pii/src/main/java/io/kairo/security/compliance/ComplianceReport.java`
- `kairo-security-pii/src/main/java/io/kairo/security/compliance/ComplianceReportCollector.java`

Implements **the existing `Consumer<KairoEvent>` shape** that
`KairoEventBus.subscribe(...)` already accepts (no new abstraction).

**Aggregation:**

| Bus event                                        | Counter                                |
|---|---|
| `DOMAIN_SECURITY` `GUARDRAIL_ALLOW`              | `guardrailAllow`                       |
| `DOMAIN_SECURITY` `GUARDRAIL_DENY`               | `guardrailDeny`                        |
| `DOMAIN_SECURITY` `GUARDRAIL_MODIFY`             | `guardrailModify`                      |
| `DOMAIN_SECURITY` `GUARDRAIL_WARN`               | `guardrailWarn`                        |
| `DOMAIN_SECURITY` `MCP_BLOCK`                    | `mcpBlock`                             |
| `DOMAIN_SECURITY` `*` w/ attr `policyName=X`     | `guardrailByPolicy[X]`                 |
| `DOMAIN_SECURITY` `GUARDRAIL_MODIFY` policy=`pii-redaction` | `piiRedactions` (cross-cut) |
| `DOMAIN_EXECUTION` `*` w/ attr `toolName=Y`      | `toolInvocations[Y]`                   |
| Other domains                                    | Ignored (forward-compatible)           |

**Thread safety:** `AtomicLong` for scalars, `ConcurrentHashMap<String, AtomicLong>`
for keyed counters — safe under reactive bus dispatch.

**Output:** `ComplianceReport.renderMarkdown()` emits a 5-section single-page
audit-trail document — header (run window in ISO-8601 UTC), guardrail decisions
table, PII redactions total, tool invocations table, guardrail decisions by
policy. Empty-state messages where appropriate. Suitable as evidence in
GDPR/HIPAA audit bundles.

**Tests:** `ComplianceReportCollectorTest` — 6/6 green (counter increments,
PII separation from generic MODIFY, tool aggregation, unknown-domain ignore,
full Markdown render, empty-state render).
