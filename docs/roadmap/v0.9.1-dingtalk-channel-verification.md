# v0.9.1 — DingTalk Channel Adapter — Verification

**State:** Released
**Scope frozen at:** 2026-04-24
**Batch:** v0.10.2 → v0.9.1 → v1.0.0-RC1
**Non-goal (v0.9 GA):** shipping a concrete IM adapter — the Channel SPI shipped
transport-free by design. v0.9.1 is the CD-train that proves the SPI against a
real transport.

## Goal

Ship DingTalk as the first concrete `Channel` implementation on top of the
v0.9.0 Channel SPI — verifier, mapper, outbound client, channel, webhook
controller, TCK extension, demo — so application teams can wire a DingTalk
custom-bot round-trip end-to-end with the Spring Boot starter.

## Scope & Evidence (single P0 item)

### B2.1 — Module scaffold

- New reactor module `kairo-channel-dingtalk` (artifactId
  `io.github.captaingreenskin:kairo-channel-dingtalk`) depending on `kairo-api`
  + `kairo-channel` + `reactor-core` + `jackson-databind` + `slf4j-api`.
- New starter module `kairo-spring-boot-starter-channel-dingtalk` depending on
  `kairo-spring-boot-starter-channel` + `kairo-channel-dingtalk` +
  `spring-boot-starter-web`.
- `kairo-bom` publishes both new artifacts.
- Reactor now has **25 modules** (was 23 at v0.10.2).

### B2.2 — Production surface

- `DingTalkSignatureVerifier` — HMAC-SHA256 over `timestamp\nsigningSecret`,
  1-hour default replay window, constant-time comparison, exposes `sign(long)`
  for reuse in outbound URI signing.
- `DingTalkMessageMapper` — `DingTalk JSON ↔ ChannelMessage`. Attribute keys
  `dingtalk.msgId` / `.msgType` / `.conversationId` / `.senderId` / `.senderNick`
  / `.sessionWebhook`. Falls back to `senderId` as destination when
  `conversationId` is absent; rejects payloads with neither.
- `DingTalkOutboundClient` — JDK `HttpClient` POST + JSON body; response
  classifier maps `errcode == 0` → ok, `errcode ∈ {130101, 130102, 130103}` → 
  `RATE_LIMITED`, other `errcode` → `REJECTED`, HTTP 429 → `RATE_LIMITED`,
  other non-2xx → `DELIVERY_FAILED`, IO failure → `SEND_FAILED`.
- `DingTalkChannel` — implements `Channel`; in-memory `ConcurrentHashMap.newKeySet()`
  dedup by DingTalk `msgId` so replayed webhook deliveries are coalesced; handler
  sees each id at most once; replay surfaces synthetic `ChannelAck.ok`.

### B2.3 — Spring Boot starter

- `@AutoConfiguration` opt-in via `kairo.channel.dingtalk.enabled=true`
  (mirrors `ChannelAutoConfiguration` pattern).
- `DingTalkProperties`: `channelId` / `webhookUrl` / `signingSecret` / `atMobiles`
  / `outboundTimeout` (5 s default) / `replayWindow` (1 h default).
- `DingTalkWebhookController @PostMapping /kairo/channel/dingtalk/callback` —
  verifies `timestamp` + `sign` headers before dispatching; HTTP 401 on
  mismatch without invoking the handler; HTTP 200 + ok envelope on success.
- `@ConditionalOnBean(ChannelInboundHandler.class)` gate: no handler → no
  channel bean (deny-safe, matches the v0.9 starter pattern).
- `spring.factories` replacement at
  `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`.

### B2.4 — TCK + extended scenarios

- `DingTalkChannelTckTest` extends `ChannelTCK` — all 3 baseline scenarios pass
  (inbound → handler → ack, outbound transport failure, per-destination ordering).
  `simulateInbound` hands the `ChannelMessage` straight into `dispatchInbound`
  so the TCK's record-equality assertions stay well-defined (the mapper's
  round-trip is covered by `DingTalkMessageMapperTest`).
- `DingTalkChannelExtendedScenariosTest` covers the 3 DingTalk-specific
  scenarios from the plan **plus** one additional DingTalk-native
  classification:
  1. Duplicate webhook delivery with the same `msgId` is deduplicated — the
     handler sees it once; replay surfaces a synthetic ok ack.
  2. Signature mismatch at the verifier layer is rejected without invoking
     the handler (`DingTalkWebhookController`-style verify-then-dispatch
     gate in the test).
  3. Outbound HTTP 429 → `ChannelFailureMode.RATE_LIMITED`.
  4. Outbound HTTP 200 + DingTalk `errcode=130101` → `RATE_LIMITED` (ensures
     the "200 OK but rate-limited" case is still classified correctly).
- `DingTalkChannelAutoConfigurationTest` — 4 Spring context tests cover
  disabled-by-default, enabled-without-handler (verifier wired, channel not),
  enabled-with-handler (full chain wired), missing-signing-secret fail-fast.

### B2.5 — Demo

- `kairo-examples/src/main/java/io/kairo/examples/dingtalk/DingTalkBotDemo.java`
  shows the programmatic surface: build the channel, register a handler that
  echoes a reply through `channel.sender()`, then simulate an inbound JSON
  payload via `dispatchInbound(String)`. Env-var driven so the demo compiles
  without real credentials.

## Out of Scope (deferred)

- Persistent de-dup store — in-memory `ConcurrentHashMap.newKeySet()` is
  bounded only by channel lifetime. Applications that need cross-restart
  idempotency can replace `DingTalkChannel` with a subclass that swaps the
  dedup set for a Redis/JDBC-backed implementation. Not a v0.9 concern.
- Feishu / Slack / generic-webhook adapters — same SPI, separate modules on
  the v0.9.x train or post-v1.0.
- Binary attachments, reactions, reply threading — adapters stuff them into
  `ChannelIdentity.attributes` until a v1.x contract lands.
