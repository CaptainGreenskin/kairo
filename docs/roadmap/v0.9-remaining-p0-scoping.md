## 2) P0 #1 — Channel SPI

### Purpose

Today, if someone wants DingTalk / 飞书 / Slack / a custom webhook to trigger an Agent (or an Expert Team), they wire it by hand in application code. A Channel SPI turns that into a contract:

```
inbound  : ChannelMessage  → AgentInvoker / TeamCoordinator
outbound : AgentResponse   → ChannelSender
```

### Scope (IN)

- `kairo-api/src/main/java/io/kairo/api/channel/` with:
  - `Channel` — SPI marker + lifecycle (`start() / stop()`).
  - `ChannelInboundHandler` — invoked when an external system sends a message to Kairo.
  - `ChannelOutboundSender` — called by Kairo to push an Agent/Team reply back to the external system.
  - Value types: `ChannelMessage`, `ChannelIdentity`, `ChannelAck`, `ChannelFailureMode`.
- New module `kairo-channel` with:
  - Default `NoopChannel` + in-memory `LoopbackChannel` (for tests / demos).
  - `ChannelRegistry` for looking up a channel by id.
- Starter: `kairo-spring-boot-starter-channel` — `@ConditionalOnProperty("kairo.channel.enabled"=true)`, auto-wires registered `Channel` beans.
- TCK (`io.kairo.channel.tck.ChannelTCK`) with a minimum scenario set:
  1. inbound message → Agent invocation → outbound reply
  2. outbound failure is surfaced as `ChannelFailureMode.SEND_FAILED`
  3. concurrent inbound messages do not lose ordering per `ChannelIdentity`

### Scope (OUT)

- No concrete IM adapter in this cycle (DingTalk / 飞书 / Slack ship as **separate modules** in v0.9.x or v1.0, consuming this SPI).
- No persistent session binding for channel identities (that becomes a v1.0 consideration once we have usage feedback).

### Deliverables

- 1 SPI surface in `kairo-api`.
- 1 module `kairo-channel` with `LoopbackChannel` + `ChannelRegistry` + TCK.
- 1 starter `kairo-spring-boot-starter-channel`.
- 1 ADR: `ADR-017-channel-spi.md`.

### Exit Criteria

- TCK's 3 scenarios pass.
- Starter auto-config test set (default-off, opt-in, user-override, missing-class) passes.
- `kairo-api.channel.*` does not reference `io.kairo.core.*` / `io.kairo.multiagent.*` / `io.kairo.expertteam.*`.
- `mvn clean verify` green.

### Rough Effort

Medium (SPI design + 1 ref impl + TCK + starter).

## 3) P0 #2 — Event Stream

### Purpose

`KairoEventBus` today is an **in-process** facade. External dashboards / frontends / ops tools can only consume events by linking against Kairo. The Event Stream gate exposes the bus over **SSE** (and optionally WebSocket) so clients can subscribe by domain / event type / correlation id without JVM coupling.

### Scope (IN)

- `kairo-api/src/main/java/io/kairo/api/event/stream/` — **transport-agnostic** SPI:
  - `EventStreamFilter` — composable predicate (by domain, eventType, attribute match).
  - `EventStreamSubscription` — `Flux<KairoEvent>` handle with cancel.
  - `KairoEventStreamAuthorizer` — deny-safe authorization SPI; default throws unless overridden.
  - Value types: `EventStreamSubscriptionRequest`, `BackpressurePolicy { BUFFER_DROP_OLDEST, BUFFER_DROP_NEWEST, ERROR_ON_OVERFLOW }`.
- **Core module** `kairo-event-stream` — transport-agnostic plumbing:
  - `EventStreamRegistry` — tracks active subscriptions, enforces a global subscriber cap.
  - `FluxBackpressureGuard` — applies the configured `BackpressurePolicy` + emits `stream_backpressure_dropped` counter.
  - Does **not** depend on Spring WebFlux / WebSocket / servlet API.
- **Transport module** `kairo-event-stream-sse` — `RouterFunction` binding `GET /kairo/events/stream` to a filtered `Flux<KairoEvent>` rendered as `text/event-stream`.
- **Transport module** `kairo-event-stream-ws` — Spring WebFlux `WebSocketHandler` binding `/kairo/events/ws` with a JSON control frame (`{"subscribe":{"domains":[...]}}`) that maps to an `EventStreamFilter`, and an outbound `KairoEvent` JSON frame stream.
- Starter: `kairo-spring-boot-starter-event-stream` — single starter, two independent toggles:
  - `kairo.event-stream.sse.enabled=true` → wires SSE endpoint.
  - `kairo.event-stream.ws.enabled=true` → wires WS handler.
  - Neither defaults to true; installing the starter alone emits no endpoints.
- Security: every enabled transport requires a `KairoEventStreamAuthorizer`. If none is supplied, startup fails with a clear error (same shape as MCP deny-safe default).

### Scope (OUT)

- No historical replay / durable storage of events (v0.9 is live-only; persistence is a v1.0 story if demanded).
- No gRPC transport (can slot in as a third transport module post-GA — the core SPI is already transport-agnostic).

### Deliverables

- 1 SPI surface in `kairo-api` (`EventStreamFilter`, `EventStreamSubscription`, `EventStreamSubscriptionRequest`, `BackpressurePolicy`, `KairoEventStreamAuthorizer`).
- 1 core module `kairo-event-stream` (registry + backpressure guard + shared value types).
- 2 transport modules: `kairo-event-stream-sse`, `kairo-event-stream-ws`.
- 1 starter `kairo-spring-boot-starter-event-stream` with independent SSE / WS toggles.
- 1 ADR: `ADR-018-event-stream.md` (covers transport-agnostic SPI + two transports + authorizer).

### Exit Criteria

- SSE integration test: Spring Boot app boots with `sse.enabled=true` + authorizer bean; `curl -N http://.../kairo/events/stream?domain=execution` receives `KairoEvent` JSON envelopes, filtered.
- WS integration test: `ReactorNettyClient` connects to `/kairo/events/ws`, sends a subscribe control frame, receives matching JSON frames; disconnect cleans up the subscription from `EventStreamRegistry`.
- Back-pressure test: slow consumer (either transport) triggers buffer overflow, counter increments, upstream `KairoEventBus` publisher is not blocked.
- Security test: without a `KairoEventStreamAuthorizer` bean, startup fails fast; with a bean that denies, both transports return 403 (SSE) / close frame 4403 (WS).
- Module boundary test: `kairo-event-stream` core does not depend on `spring-webflux` or `reactor-netty`; those are only in the transport modules.
- `mvn clean verify` green.

### Rough Effort

Medium (two transports + shared back-pressure correctness + authorizer wiring). Extra submodule split is modest overhead vs. the optionality it buys.

## 4) P0 #3 — OTel Security Exporter

### Purpose

`kairo-observability` already has `OTelTracer` / `GenAiSemanticAttributes`. This P0 closes the gap for **security domain events** — Guardrail decisions, MCP deny events, policy violations — by exporting them as OpenTelemetry spans/logs so they flow into SIEMs and APM backends through one standard pipe.

### Scope (IN)

- New component in `kairo-observability`:
  - `KairoEventOTelExporter` — listens to `KairoEventBus` for the configured domain set, converts each envelope to an OTel `LogRecord` (preferred over Span since Kairo events are point-in-time, not duration-bound).
  - Domain-specific attribute mapping:
    - `security` → `kairo.security.decision`, `kairo.security.guardrail.id`, `kairo.security.violation.kind`
    - `execution` → `kairo.execution.phase`, `kairo.execution.tool.id`, `kairo.execution.iteration`
    - `team` → `kairo.team.status`, `kairo.team.step.id`, `kairo.team.evaluator.verdict`
  - Plus all `KairoEvent.attributes` namespaced under `kairo.<domain>.*`.
  - Sampling: always-on for `security`; configurable `sampling-ratio` for `execution`/`team` (default 1.0) since those are higher-volume.
- Starter hook in `kairo-spring-boot-starter-observability`:
  - `@ConditionalOnProperty("kairo.observability.otel-exporter.enabled"=true)` — off by default.
  - `kairo.observability.otel-exporter.include-domains` — default `[security]`; user sets e.g. `security,execution,team` to widen.
  - `kairo.observability.otel-exporter.sampling-ratio` — default `1.0`.
  - Auto-subscribes to `KairoEventBus`, wires into the existing OTel SDK `LoggerProvider`.

### Scope (OUT)

- No new OTel SDK version bump (keep current).
- No custom SIEM adapter — we export via OTel, downstream routing is the SIEM's job.
- No retention / archival — the SIEM owns that.

### Deliverables

- `KairoEventOTelExporter` implementation in `kairo-observability`.
- `OTelExporterProperties` with `enabled` / `includeDomains` / `samplingRatio` / `attributeRedactionPatterns`.
- Starter auto-config in `kairo-spring-boot-starter-observability`.
- Docs: extend `docs/en/guide/security-observability-schema.md` with the attribute schema for all three domains.
- 1 ADR: `ADR-019-kairo-event-otel-exporter.md`.

### Exit Criteria

- Unit test: publishing a synthetic envelope per domain (`security` / `execution` / `team`) results in 1 OTel `LogRecord` each with the mapped attributes under the correct `kairo.<domain>.*` namespace.
- Domain-filter test: with `include-domains=[security]`, `execution`/`team` envelopes are ignored.
- Sampling test: `samplingRatio=0.0` yields zero records for `execution`; `samplingRatio=1.0` yields all; `security` is always-on regardless.
- Integration test: Spring Boot app + in-memory `LogRecordExporter` captures records end-to-end.
- Starter auto-config tests: default-off, opt-in wires, user-provided `KairoEventOTelExporter` bean wins (`@ConditionalOnMissingBean`).
- `mvn clean verify` green.

### Rough Effort

Medium (three-domain attribute mapping + sampling + starter glue).

## 5) Dependencies and Execution Order

```
P0#2 Event Stream ─────┐
                        ├──► P0#1 Channel SPI (optional but recommended: Channel can reuse EventStreamFilter concept for observability)
P0#3 OTel Security ───┘
```

- P0#2 and P0#3 are **independent** and can be done in parallel (both hang off `KairoEventBus`).
- P0#1 **Channel SPI** can be done in parallel with both — but if we want a compelling "DingTalk → Expert Team" demo for v1.0 launch, Channel is the critical path.

**Recommended order**: P0#2 Event Stream → P0#3 OTel Security Exporter → P0#1 Channel SPI.
Reason: Event Stream and OTel both validate the `KairoEventBus` design under external-consumer pressure before we freeze the SPI for v1.0; Channel SPI builds on that validated event flow.

## 6) Locked Decisions (2026-04-24)

| # | Decision | Choice |
|---|---|---|
| D1 | Channel SPI: ship concrete IM adapter in v0.9? | **(a)** SPI + `LoopbackChannel` only. Real IM adapters (DingTalk / 飞书 / Slack / webhook) ship as separate modules in the v0.9.x continuous-delivery train, each consuming the frozen SPI. |
| D2 | Event Stream: transport | **(b)** SSE + WebSocket. Both land in v0.9 GA so the SPI is validated against the two dominant streaming shapes before v1.0 freeze. Shared core (`EventStreamFilter`, back-pressure, authorizer) is transport-agnostic; each transport sits behind its own `@ConditionalOnProperty`. |
| D3 | OTel exporter: domain scope | **(b)** `security` + `execution` + `team` flagged. Default remains `security`-only (deny-safe audit story); the property `kairo.observability.otel-exporter.include-domains` opts additional domains in. |
| D4 | Release packaging | **(a)** v0.9.0 GA ships all three P0s together. No v0.9.1/v0.9.2 patch fan-out. |
| D5 | Deprecated API deletion | **Delete in v0.9 cycle.** `io.kairo.api.task.*` and `TeamScheduler` (already `@Deprecated(forRemoval=true)` since v0.10) are physically removed before v0.9 GA so v1.0 opens with a clean API surface. |

Consequence of D2 raising WebSocket into GA scope: P0#2 grows from Small-Medium to **Medium**; factor shared SSE/WS plumbing into a dedicated `kairo-event-stream` core + `kairo-event-stream-sse` + `kairo-event-stream-ws` submodule layout (see §3 below — now refactored).

Consequence of D3 widening to execution/team: exporter renamed from `SecurityEventOTelExporter` to `KairoEventOTelExporter` with a domain filter; the single-purpose naming would have lied about its scope.
